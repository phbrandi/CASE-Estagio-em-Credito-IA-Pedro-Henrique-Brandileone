1. VISÃO GERAL
================================================================================

O sistema é um pipeline automatizado de monitoramento de crédito para 15 empresas
brasileiras listadas em bolsa. A cada execução, ele coleta notícias e comunicados
públicos das últimas 26 semanas (180 dias), classifica cada item com um modelo de
linguagem (LLM), e entrega dois outputs prontos para uso por um gestor de crédito:
um CSV estruturado com 12 colunas (news.csv) e um relatório PDF com gráficos
analíticos (monitor.pdf).

Empresas monitoradas:
OceanPact (OPCT3), Brava Energia (BRAV3), PetroRio (PRIO3), PetroReconcavo (RECV3),
NTS (NTSB11), Aegea (AGYS3), Equatorial (EQTL3), Copasa (CSMG3), Cosan (CSAN3),
Vamos (VAMO3), Mills (MILS3), Armac (ARML3), BTG (BPAC11), Unipar (UNIP6),
Multiplan (MULT3).


2. FERRAMENTAS UTILIZADAS
================================================================================

COLETA E EXTRAÇÃO WEB
  - feedparser         : parse do feed RSS do Google News — retorna título, link,
                         data de publicação (published_parsed como struct_time)
                         e resumo HTML de cada notícia sem autenticação.
  - requests           : requisições HTTP para raspagem das páginas de RI.
  - BeautifulSoup/lxml : parse do HTML das páginas de RI; localiza links com
                         heurísticas de palavras-chave (fato relevante, earnings,
                         dividendo, etc.) e extrai datas de atributos como
                         data-date, datetime, ou texto próximo via regex.
  - Playwright         : browser headless (Chromium) controlado via Python assíncrono.
                         Usado para seguir os redirecionamentos JavaScript do Google
                         News e extrair o texto real dos artigos. Roda até 10 abas
                         em paralelo via asyncio.Semaphore.

PROCESSAMENTO
  - pandas             : manipulação e escrita do CSV de saída.
  - difflib            : SequenceMatcher para deduplicação por similaridade de título
                         (threshold 0.85).
  - hashlib/SHA256      : deduplicação por URL normalizada.

IA / LLM
  - Anthropic API      : modelo claude-haiku-4-5-20251001, chamado via SDK Python.
                         Recebe título + trecho (até 1.500 caracteres) e retorna
                         JSON estruturado com sentimento, tags, severidade e resumo.

RELATÓRIO
  - matplotlib/seaborn : geração dos 4 gráficos PNG (ranking, heatmap, tendência,
                         top tags). Backend Agg (sem GUI) para compatibilidade
                         com Windows sem display.
  - ReportLab          : composição do PDF de monitoramento com os gráficos e
                         análise executiva em texto.

INFRAESTRUTURA
  - python-dotenv      : leitura da ANTHROPIC_API_KEY do arquivo .env, sem
                         expor credenciais no código.
  - logging            : dois handlers simultâneos — arquivo logs/run.log e
                         terminal — com timestamp em cada linha.


3. PIPELINE: INPUT → ETAPAS → OUTPUTS
================================================================================

INPUT
-----
  companies.csv — cadastro das 15 empresas com campos:
    empresa, ticker, setor, segmento, ri_url

ETAPA 1 — COLETA (collector.py :: buscar_links)
-------------------------------------------------
Para cada empresa, duas fontes são consultadas em paralelo:

  a) Google News RSS
     URL montada dinamicamente:
       https://news.google.com/rss/search?q={nome}+{ticker}&hl=pt-BR&gl=BR
     feedparser parse o feed e retorna até ~100 itens. Para cada entrada:
       - Data extraída de published_parsed (struct_time UTC — mais confiável
         que a string published, que frequentemente vinha em formato não-padrão
         e causava parse silencioso com fallback para datetime.now()).
       - Itens com data anterior a 180 dias são descartados imediatamente.
       - Resumo HTML do RSS é limpo com BeautifulSoup e guardado como
         snippet_ou_trecho inicial (substituído depois pelo Playwright).

  b) Página de RI (Relações com Investidores)
     GET na ri_url de cada empresa. O HTML é parseado e todos os links <a>
     são filtrados por heurística: texto do link contém palavras-chave de RI
     (fato relevante, comunicado, resultado, dividendo, debenture, etc.) OU
     o href termina em .pdf. Para cada link relevante:
       - URL absoluta montada (resolve caminhos relativos).
       - Data extraída de atributos da tag (data-date, datetime, title) ou
         regex no texto do elemento pai. Fallback: "desconhecida".

ETAPA 2 — DEDUPLICAÇÃO (collector.py :: deduplicar)
----------------------------------------------------
Executada ANTES da extração de snippets para economizar requisições.
Duas passagens:
  Passagem 1 — Hash SHA256 da URL normalizada (sem query string, lowercase).
               Remove duplicatas exatas de URL, como o mesmo artigo indexado
               com e sem parâmetros de tracking.
  Passagem 2 — SequenceMatcher (difflib) com threshold 0.85 nos títulos.
               Remove variações do mesmo artigo republicadas com título
               ligeiramente diferente em veículos distintos.

ETAPA 3 — EXTRAÇÃO DE SNIPPETS (collector.py :: extrair_snippets_playwright)
---------------------------------------------------------------------------
O Playwright sobe um browser Chromium headless e navega até cada URL.
Até 10 páginas são abertas em paralelo via asyncio.gather + Semaphore.
Para cada página:
  1. Aguarda o carregamento do DOM (domcontentloaded).
  2. Remove via JavaScript elementos de navegação, rodapé, anúncios e banners.
  3. Tenta extrair o texto de <article>; se ausente, concatena parágrafos <p>
     com mais de 80 caracteres.
  4. Retorna até 1.500 caracteres do texto resultante.
Sites com paywall, anti-bot ou que retornam erro retornam string vazia —
o item ainda entra no pipeline mas o resumo será gerado só a partir do título.

ETAPA 4 — CLASSIFICAÇÃO LLM (classifier.py :: classify_llm)
------------------------------------------------------------
Para cada item, uma chamada à API Anthropic com:
  - Modelo: claude-haiku-4-5-20251001 (rápido e econômico)
  - Input: título (até 300 chars) + trecho (até 1.500 chars)
  - Output esperado: JSON com 4 campos (ver seção 5 — Prompts)
  - Rate limiting: sleep de 0.5s entre chamadas.
  - Retry: se o JSON retornado for inválido ou falhar validação, uma segunda
    chamada é feita. Se a segunda também falhar, defaults são aplicados
    (neutro, severidade 1, tag setor_macro).

ETAPA 5 — SALVAMENTO INCREMENTAL (run.py :: main)
--------------------------------------------------
Após cada empresa processada, o CSV parcial é salvo em output/news.csv.
Isso garante que uma falha na empresa 12, por exemplo, não perde o trabalho
já feito nas 11 anteriores.

ETAPA 6 — RELATÓRIO (monitor.py :: gerar_relatorio)
----------------------------------------------------
Lê o news.csv final e gera:
  - ranking_atencao.png : empresas ordenadas por score de atenção
                          (score = negativas×2 + severidade_3×3)
  - heatmap.png         : eventos negativos por empresa × mês
  - tendencia.png       : evolução mensal do sentimento no portfólio
  - top_tags.png        : frequência das tags temáticas
  - monitor.pdf         : relatório com os 4 gráficos e análise executiva

OUTPUTS
-------
  output/news.csv    — 902 itens classificados, 12 colunas
  output/monitor.pdf — relatório PDF para o gestor
  output/charts/     — 4 gráficos PNG
  logs/run.log       — log completo com timestamp de cada ação


4. WORKFLOW EM LOTE — COMO O PIPELINE RODA PARA TODAS AS EMPRESAS
================================================================================

O orquestrador run.py itera sobre as 15 linhas do companies.csv em sequência.
Para cada empresa o loop é:

  for idx, empresa in enumerate(empresas, start=1):
      items = buscar_links(empresa)        # etapas 1
      items = deduplicar(items)            # etapa 2
      extrair_snippets_playwright(items)   # etapa 3
      for item in items:
          classificacao = classify_llm(item) # etapa 4
          item.update(classificacao)
      salvar_csv_parcial()                 # etapa 5

Todas as exceções de rede e de API são capturadas com try/except por empresa.
Se uma empresa falha (timeout, site fora do ar, erro de API), o pipeline
registra o erro no log, pula para a próxima e não interrompe a execução.

A extração de snippets dentro de cada empresa é assíncrona (até 10 threads
paralelas via asyncio). A classificação LLM é sequencial por item para
respeitar o rate limit da API.

6. PROMPTS E SAÍDA ESTRUTURADA
================================================================================

PROMPT PRINCIPAL (enviado ao Claude Haiku para cada item)
---------------------------------------------------------

  "Você é um analista de crédito sênior. Classifique a notícia abaixo SOMENTE
  com base no texto fornecido. NÃO infira nada além do que está explícito.

  Título: {titulo}
  Trecho: {trecho}

  Retorne APENAS um JSON válido com exatamente este schema (sem texto adicional,
  sem markdown, sem backticks):
  {
    "sentimento": "positivo|neutro|negativo",
    "tags": ["tag1", "tag2"],
    "severidade": 1,
    "resumo_curto": "1-2 frases informativas sobre o evento"
  }

  Regras:
  - sentimento: escolha exatamente um entre positivo, neutro ou negativo
  - tags: lista com 1 ou mais dos valores: liquidez_refinanciamento,
    resultado_guidance, governanca, legal_regulatorio, operacional_incidente,
    m&a_estrategia, setor_macro, esg_reputacional
  - severidade: 1 (baixo impacto/ruído), 2 (relevante, merece atenção),
    3 (potencialmente material)
  - resumo_curto: 1-2 frases que informem o leitor de forma objetiva. Inclua
    obrigatoriamente dados concretos presentes no trecho: valores em R$ ou USD,
    percentuais, volumes, datas, nomes de partes envolvidas, ratings. Se o trecho
    não tiver dados quantitativos, descreva o fato central com o máximo de
    especificidade possível. NUNCA escreva frases vagas como 'a empresa anunciou
    resultados' — diga o que foram os resultados.

  Retorne SOMENTE o JSON, sem nenhum texto antes ou depois."

EXEMPLOS DE SAÍDA GERADA PELO MODELO
-------------------------------------

  Exemplo 1 — com dado quantitativo concreto:
    Título : "OceanPact (OPCT3) anuncia fusão com CBO e cria empresa de apoio
              marítimo com R$ 13,6 bi em contratos"
    Resumo : "OceanPact (OPCT3) anuncia fusão com CBO para criar empresa de apoio
              marítimo com R$ 13,6 bilhões em contratos. A operação representa
              expansão estratégica no segmento de serviços marítimos."
    Sentimento: positivo | Severidade: 2 | Tags: m&a_estrategia

  Exemplo 2 — recompra de ações com percentual:
    Título : "OceanPact aprova combinação com CBO e anuncia recompra de ações
              de até 3% do capital"
    Resumo : "OceanPact aprovou combinação com CBO e anunciou programa de
              recompra de ações de até 3% do capital. A operação representa
              movimento estratégico de consolidação e retorno de capital."
    Sentimento: positivo | Severidade: 2 | Tags: m&a_estrategia

  Exemplo 3 — aviso legal sem dados concretos (modelo descreve o fato):
    Título : "Notas Legais"
    Resumo : "Multiplan divulga aviso sobre declarações prospectivas em seu
              website de RI, alertando que resultados futuros podem diferir
              das projeções devido a riscos e incertezas. Nota legal padrão
              de disclaimer sem dados quantitativos."
    Sentimento: neutro | Severidade: 1 | Tags: legal_regulatorio;governanca

COMO A SAÍDA ESTRUTURADA É GARANTIDA
--------------------------------------

  1. Schema explícito no prompt — o modelo recebe o JSON de exemplo e é
     instruído a não incluir texto fora dele.

  2. Limpeza do retorno — a função _limpar_json() remove blocos de markdown
     (```json ... ```) e extrai apenas o conteúdo entre o primeiro { e o
     último } caso o modelo adicione prefácio acidental.

  3. Validação programática — _validar() confere:
       - sentimento ∈ {"positivo", "neutro", "negativo"}
       - tags é lista não-vazia onde cada elemento ∈ TAGS_VALIDAS (8 valores)
       - severidade ∈ {1, 2, 3}
       - resumo_curto é string não-vazia

  4. Retry com fallback — se a validação falha, uma segunda chamada é feita.
     Se a segunda também falha, defaults conservadores são aplicados
     (neutro, severidade 1, ["setor_macro"]) sem interromper o pipeline.

  5. CSV com schema fixo — run.py garante que as 12 colunas existam e estejam
     na ordem correta antes de salvar, preenchendo com string vazia se alguma
     coluna estiver ausente.


7. LIMITAÇÕES E ESTRATÉGIAS CONTRA ALUCINAÇÃO
================================================================================

LIMITAÇÃO 1 — GOOGLE NEWS USA REDIRECIONAMENTO JAVASCRIPT
----------------------------------------------------------
O Google News não entrega a URL final diretamente: os links do RSS apontam
para um redirect (/redir/...) que só é resolvido via JavaScript no browser.
A biblioteca requests não executa JavaScript e portanto não conseguia chegar
ao artigo original.

Solução implementada: Playwright (browser headless Chromium) executa o
JavaScript, segue o redirect e acessa o conteúdo real da página de destino.
Isso viabiliza a extração do texto completo do artigo como contexto para o LLM.

LIMITAÇÃO 2 — SITES COM PAYWALL OU ANTI-BOT
--------------------------------------------
Alguns veículos (Valor Econômico, Bloomberg, Exame Premium) bloqueiam
scrapers mesmo com Playwright. Nesses casos snippet_ou_trecho retorna vazio
e o Claude classifica apenas com base no título. O item é mantido no CSV
com o campo em branco — o link original está sempre presente, permitindo
ao analista consultar a fonte manualmente.

LIMITAÇÃO 3 — DATAS AUSENTES EM PÁGINAS DE RI
----------------------------------------------
A maioria das páginas de RI brasileiras não expõe datas em atributos
estruturados do HTML. O código tenta extrair de data-date, datetime, title
e via regex no texto do elemento pai. Quando nenhuma estratégia funciona,
o campo data_publicacao recebe "desconhecida" — nunca uma data inventada.

LIMITAÇÃO 4 — EMPRESAS SEM PRESENÇA RELEVANTE NO GOOGLE NEWS
-------------------------------------------------------------
Empresas menores ou emissoras de renda fixa (ex: NTS) têm pouca cobertura
jornalística. O pipeline coleta 0 itens nesses casos e registra no log,
sem erro. A solução de longo prazo seria adicionar fontes alternativas
(EDGAR, CVM, Bloomberg BVMF).

ESTRATÉGIAS CONTRA ALUCINAÇÃO DO LLM
--------------------------------------

  a) Classificação restrita ao trecho fornecido
     O prompt instrui explicitamente: "Classifique SOMENTE com base no texto
     fornecido. NÃO infira nada além do que está explícito." Isso reduz o
     risco de o modelo extrapolar sentimentos a partir de conhecimento prévio
     sobre a empresa.

  b) Vocabulário controlado (enum fechado)
     sentimento tem apenas 3 valores válidos; tags têm apenas 8 opções;
     severidade aceita apenas 1, 2 ou 3. Qualquer valor fora desse conjunto
     é rejeitado pela validação e leva a retry/fallback.

  c) URL sempre preservada
     Cada item carrega a URL de origem. O gestor pode verificar qualquer
     classificação acessando a fonte primária — o modelo nunca é a única
     fonte de verdade.

  d) resumo_curto ancorado em dados do trecho
     O prompt exige que o resumo cite dados concretos presentes no texto
     (valores, percentuais, nomes). Isso incentiva o modelo a reproduzir
     informações verificáveis em vez de gerar generalizações vagas.

  e) Fallback conservador
     Quando duas tentativas de classificação falham, o item recebe
     sentimento "neutro" e severidade 1 (menor grau de alerta). Isso é
     propositalmente conservador: preferível subestimar do que gerar um
     alerta falso de risco alto.

8. INFORMAÇÕES ADICIONAIS E EXTENSÕES POSSÍVEIS
================================================================================

QUALIDADE DA COBERTURA POR FONTE
---------------------------------
A principal fonte de volume é o Google News RSS (~94% dos itens), que agrega
artigos de InfoMoney, Valor Econômico, Exame, Reuters Brasil, entre outros.
As páginas de RI contribuem com comunicados oficiais (fatos relevantes, atas,
avisos ao mercado) que raramente aparecem no Google News — são 52 itens (6%),
mas de alta relevância para análise de crédito por serem documentos primários.

CUSTO ESTIMADO DE EXECUÇÃO
---------------------------
Com claude-haiku-4-5-20251001, cada chamada consume em média ~600 tokens de
input e ~150 tokens de output. Para 902 itens:
  Input  : ~540.000 tokens
  Output : ~135.000 tokens
Com a precificação atual do Haiku ($0,80/Mtok input, $4,00/Mtok output),
o custo total da classificação de um portfólio completo fica em torno de
USD 0,43 por execução — extremamente econômico para uso em produção.

TEMPO DE EXECUÇÃO
-----------------
A execução completa (15 empresas, 902 itens, coleta + dedupe + snippets + LLM)
levou aproximadamente 1h40min na última execução registrada. O gargalo é a
classificação sequencial item a item (rate limit da API). Com paralelização
de chamadas LLM (batch API da Anthropic), esse tempo poderia ser reduzido
em ~70%.

EXTENSÕES POSSÍVEIS PARA VERSÕES FUTURAS
-----------------------------------------

  1. Alertas automáticos por e-mail ou Slack
     Um módulo de alertas poderia filtrar itens com severidade 3 e negativo
     e enviar uma notificação ao gestor assim que o pipeline terminar, sem
     precisar abrir o PDF.

  2. Banco de dados em vez de CSV
     Substituir news.csv por um banco SQLite ou PostgreSQL permitiria consultas
     históricas, rastreio de evolução de sentimento por empresa ao longo do
     tempo, e evitaria re-processar itens já classificados em execuções anteriores.

  3. Fonte de dados CVM/EDGAR
     A CVM disponibiliza um sistema de busca de documentos (ENET) com API
     não-oficial. Integrar isso tornaria a cobertura de fatos relevantes e ITRs
     completa e estruturada, independente do scraping de RI.

  4. Paralelização da classificação LLM
     A Anthropic oferece uma Batch API que processa até 10.000 requisições
     assíncronas com desconto de 50% no custo e sem rate limit restritivo.
     Isso reduziria o tempo de classificação de ~90 min para ~5-10 min.

  5. Score de credibilidade da fonte
     Ponderar o sentimento pelo histórico de confiabilidade da fonte
     (veículo jornalístico vs. blog vs. comunicado oficial) tornaria o
     ranking de atenção mais preciso e menos suscetível a ruído.

  6. Detecção de recorrência de temas
     Identificar automaticamente quando um mesmo tema (ex: renegociação de
     dívida, investigação regulatória) aparece em múltiplos itens de uma
     mesma empresa em curto intervalo — sinal de que o evento é persistente
     e não pontual.

REPRODUTIBILIDADE
-----------------
O pipeline é completamente reproduzível a partir do repositório:
  1. pip install -r requirements.txt
  2. python -m playwright install chromium
  3. Criar .env com ANTHROPIC_API_KEY=sk-ant-...
  4. python scripts/run.py

Todas as dependências estão fixadas com versões explícitas no requirements.txt.
O arquivo companies.csv contém o cadastro completo das 15 empresas com as
URLs de RI verificadas.
